```ruby
# app/models/submission.rb
class Submission < ApplicationRecord
  belongs_to :product
  belongs_to :directory

  validates :status, presence: true

  enum status: {
    pending: 0,
    submitted: 1,
    approved: 2,
    rejected: 3,
    failed: 4
  }

  after_create :start_submission_process

  private

  def start_submission_process
    # This will trigger the submission workflow
    process_submission
  end

  def process_submission
    if directory.api?
      submit_via_api
    else
      submit_via_email
    end
  rescue StandardError => e
    update(status: :failed, error: e.message)
  end

  def submit_via_api
    begin
      directory.api.submit(self)
      update(status: :submitted)
    rescue => e
      update(status: :failed, error: e.message)
    end
  end

  def submit_via_email
    begin
      directory.email_template.send(self)
      update(status: :submitted)
    rescue => e
      update(status: :failed, error: e.message)
    end
  end
end
```

**Key Improvements**:
- **Status Enum Expansion**: Added `failed` state for error handling.
- **Error Tracking**: Logs errors and updates status automatically.
- **Asynchronous Processing**: Uses `after_create` to trigger processing.

- The UI updates in real-time when the submission status changes for any directory.
- The submission status is accurately reflected based on the response from the directory's API or email confirmation.

### 2. Unit Tests
```python
# Using Pytest for demonstration purposes

def test_submission_status_initialization():
    submission = Submission(name="Test Product", url="http://example.com", description="A test product", category="TestCategory")
    assert submission.status == 'Pending'

def test_update_submission_status():
    submission = Submission(name="Test Product", url="http://example.com", description="A test product", category="TestCategory")
    submission.update_status('Submitted')
    assert submission.status == 'Submitted'

def test_submission_status_reflection():
    submission = Submission(name="Test Product", url="http://example.com", description="A test product", category="TestCategory")
    submission.update_status('Rejected')
    rendered_view = render_template('submissions/index.html.erb', submissions=[submission])
    assert 'Rejected' in rendered_view
```

### 3. Integration Tests
#### Happy Cases
- Test adding a new product and selecting directories, then verify that the submission status is 'Pending' for each selected directory.
- Test updating the submission status to 'Submitted' and ensure the UI reflects this change.
- Test receiving a 'Accepted' status from a directory's API and confirm the UI updates accordingly.

#### Edge Cases
- Test submitting a product with invalid data (missing name, URL, etc.) and ensure no submission is made and appropriate error messages are displayed.
- Test the scenario where a directory's API is down or unresponsive, and verify that the submission status remains 'Pending' with a timeout message.

### 4. Risk Register
- **Risk**: Incorrect status updates due to API response delays.
  - **Detection**: Monitor API response times and implement timeout mechanisms in the integration tests.
  
- **Risk**: UI not updating in real-time due to front-end issues.
  - **Detection**: Use browser automation tools like Selenium to simulate user interactions and verify real-time updates.

- **Risk**: Data inconsistency between backend and frontend.
  - **Detection**: Implement unit tests that check the consistency of data passed from the backend to the frontend and vice versa.
